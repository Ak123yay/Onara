# ADR-003: Why Supabase

**Date**: 2026-05-14  
**Status**: Accepted  
**Deciders**: Solo founder

---

## Context

Onara needs a backend data layer covering: relational database, user authentication, file storage, and scheduled jobs. The system needs to launch fast, cost near zero in v1, and scale without re-architecture.

## Decision

Use **Supabase** as the all-in-one backend: PostgreSQL + Auth + Storage + pg_cron.

## Alternatives Considered

| Option | Rejected Because |
|--------|-----------------|
| **PlanetScale** (MySQL) | MySQL-only; no auth; no storage; no pg_cron; more moving parts |
| **Neon** (serverless Postgres) | No auth; no storage; pg_cron setup more complex |
| **Firebase** (Firestore) | NoSQL — relational data model is a poor fit; no SQL queries; vendor lock-in deeper |
| **Self-hosted Postgres** | Ops overhead; no built-in auth; no storage; too much maintenance for a solo founder |
| **PocketBase** | No pg_cron; smaller ecosystem; less documentation |

## Reasoning

**Auth built-in**: Supabase Auth handles Google OAuth + email/password with one SDK. Zero auth infrastructure to maintain.

**RLS is a security primitive**: Row Level Security at the database level is stronger than application-level checks. One mistake in application code doesn't leak data — the DB enforces isolation.

**pg_cron for scheduled jobs**: Trial email sequences (Day 11/13/14) and monthly revision resets require reliable scheduled jobs. pg_cron runs inside Postgres — no separate queue or cron service.

**Storage for HTML archives**: Each site generation needs the HTML backed up. Supabase Storage eliminates a separate S3/R2 dependency.

**Free tier is generous**: 500MB DB, 1GB storage, 50K MAU free. Covers all of v1 without cost.

**Next.js client SDK**: `@supabase/supabase-js` + `@supabase/ssr` is battle-tested with Next.js App Router. Session management is solved.

## Trade-offs Accepted

- Supabase-specific pg_cron syntax (minor lock-in)
- Managed service dependency — if Supabase has an outage, so does Onara
- `SUPABASE_SERVICE_ROLE_KEY` must never reach the client (enforced by env var naming convention)

## Consequences

- All user data lives in Supabase Postgres (7 tables, fully RLS-protected)
- Auth sessions managed by Supabase JWT — Next.js validates on every API call
- pg_cron handles 4 scheduled jobs (see `wiki/data/models.md`)
- Storage bucket `site-html` archives every generated site
